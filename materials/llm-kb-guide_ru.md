## 1. Проблема: почему обычные базы знаний не работают

Большинство людей работают с документами через LLM так: загружают файлы, модель ищет релевантные куски при каждом запросе, генерирует ответ. Это называется RAG (Retrieval-Augmented Generation).

**Проблемы RAG-подхода:**

- **Нет накопления** — каждый вопрос начинается с нуля
- **Нет синтеза** — LLM не связывает информацию между источниками
- **Нет обновления** — новые данные не корректируют старые выводы
- **Нет структуры** — знание размыто по чанкам, без перекрёстных ссылок

«NotebookLM, ChatGPT file uploads и большинство RAG-систем работают именно так. LLM переоткрывает знания с нуля при каждом вопросе.»
— Андрей Карпатый

**Типичная ситуация.** Ты собираешь статьи, заметки, транскрипты. Через месяц их 50+. Задаёшь вопрос LLM — он находит 3 из 50 релевантных чанков. Завтра похожий вопрос — другие 3 чанка. Никогда нет полной картины. Знания не накапливаются — они переизобретаются каждый раз.

---

## 2. Решение: LLM Knowledge Base

Идея другая. Вместо того чтобы искать по сырым документам при каждом запросе, LLM **инкрементально строит и поддерживает персистентную вики** — структурированную коллекцию markdown-файлов с перекрёстными ссылками.

> **Ключевое отличие:** Вики — это персистентный, компаундируемый артефакт. Перекрёстные ссылки уже там. Противоречия уже отмечены. Синтез уже отражает всё, что ты прочитал. Вики становится богаче с каждым добавленным источником и каждым заданным вопросом.

### Разделение ролей

| Ты (человек) | LLM (агент) |
|---|---|
| Курируешь источники | Суммаризация и индексирование |
| Задаёшь направление | Перекрёстные ссылки |
| Задаёшь вопросы | Обновление при новых данных |
| Думаешь над смыслом | Бухгалтерия и консистентность |

«Скучная часть поддержания базы знаний — не чтение и не мышление, а бухгалтерия. Люди бросают вики, потому что затраты на поддержание растут быстрее ценности. LLM не устают и могут обновить 15 файлов за один проход.»
— Андрей Карпатый

### Примеры применения

- **Исследования** — глубокое изучение темы недели или месяцы, накопление статей и выводов
- **Бизнес** — внутренняя вики из Slack, митингов, звонков, документов
- **Личное развитие** — цели, здоровье, заметки, подкасты
- **Чтение книг** — персонажи, темы, сюжетные линии, связи
- **Конкурентный анализ** — продукты, компании, стратегии

## 3. Архитектура: 3 слоя

### Обзор системы

| Слой | Назначение | Пишет | Читает | Правило |
|---|---|---|---|---|
| `raw/` | Сырые источники | Ты | LLM | Никогда не изменять |
| `wiki/` | LLM-генерируемая вики | LLM | Ты | LLM владеет полностью |
| `outputs/` | Отчёты, ответы | LLM | Ты | Можно вернуть в wiki |
| `CLAUDE.md` | Схема (конфигурация) | Ты + LLM | LLM | Эволюционирует со временем |

### Структура папок

```
my-knowledge-base/
├── raw/                <- Неизменяемые исходные документы
│   ├── articles/       <- Статьи, блог-посты
│   ├── papers/         <- Исследования, научные работы
│   ├── notes/          <- Личные заметки, идеи
│   ├── telegram/       <- Посты из Telegram
│   └── images/         <- Диаграммы, скриншоты
├── wiki/               <- Вики, генерируемая LLM
│   ├── index.md        <- Каталог всех страниц
│   ├── log.md          <- Хронологический лог операций
│   └── _templates/     <- Шаблоны страниц
├── outputs/            <- Отчёты, аналитика
└── CLAUDE.md           <- Схема для LLM
```

### Ключевые файлы

**`wiki/index.md`** — каталог контента. Список всех страниц с описаниями по доменам. LLM читает его первым при любом запросе, чтобы найти релевантные страницы.

**`wiki/log.md`** — хронологический лог. Append-only запись всех операций: ingest, query, lint. Формат: `## [YYYY-MM-DD] операция | описание`.

**`CLAUDE.md`** — схема системы. Говорит LLM как вики устроена, какие конвенции, какие workflow. Без него LLM — просто чатбот. С ним — дисциплинированный wiki-maintainer.

## 4. Установка и настройка

### Необходимые инструменты

| Инструмент | Для чего | Где взять |
|---|---|---|
| Obsidian | Просмотр и навигация по вики | obsidian.md |
| Claude Code | LLM-агент для поддержания вики | claude.ai/code |

### Опционально

- **Obsidian Web Clipper** — сохранение статей из браузера в `.md`
- **yt-dlp** — транскрипты с YouTube
- **qmd** (tobi) — локальный поиск по markdown (BM25 + vector search)
- **Dataview** — Obsidian-плагин для запросов по frontmatter

### Шаг 1: Создание структуры

Открой терминал в папке своего Obsidian-волта и выполни:

```bash
mkdir -p raw/articles raw/papers raw/notes raw/telegram raw/images
mkdir -p wiki/_templates outputs
```

> Совет: можно просто открыть Claude Code в папке волта и сказать: «Создай структуру LLM Knowledge Base по паттерну Карпатого». Он создаст всё за тебя.

### Шаг 2: Создание index.md и log.md

**`wiki/index.md`** — каталог всех страниц с разделами по доменам.

**`wiki/log.md`** — хронологический лог. Формат: `## [YYYY-MM-DD] операция | описание`.

### Шаг 3: Настройка CLAUDE.md

Добавь в `CLAUDE.md`:

1. **Архитектура** — описание трёх слоёв (raw, wiki, outputs)
2. **Frontmatter schema** — поля для каждой вики-страницы
3. **Конвенции именования** — kebab-case для вики, дата-префикс для статей
4. **Операции** — Ingest, Query, Lint workflow
5. **Домены интересов** — твои темы

## 5. CLAUDE.md — мозг системы

Это самый важный файл. Без него LLM — просто чатбот. С ним — **дисциплинированный wiki-maintainer**.

### Frontmatter Schema

Каждая вики-страница должна иметь такой frontmatter:

```yaml
---
type: entity | concept | project | person | summary
domain: ai | communities | infrastructure | content
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: ["[[raw/articles/filename]]"]
tags: []
---
```

### Пример вики-страницы

```markdown
# Название страницы

## Summary
1-2 абзаца обзора.

## Key Details
Структурированная информация.

## Related
- [[wiki/related-page]] --- почему связано

## Sources
- [[raw/articles/source-file]]
```

> **Важно:** CLAUDE.md эволюционирует со временем. Начни с простой версии и расширяй по мере того, как понимаешь что работает для твоего домена. Карпатый: «super simple and flat».

## 6. Три операции: Ingest, Query, Lint

### Операция 1: Ingest (Загрузка)

Ты добавил файл в `raw/`. Говори LLM:

«Обработай новый файл в `raw/articles/2026-04-06-название.md`»

**Что происходит:**

1. LLM читает источник полностью
2. Обсуждает с тобой ключевые выводы
3. Создаёт или обновляет вики-страницы в `wiki/`
4. Обновляет `wiki/index.md` и `wiki/log.md`
5. Добавляет `[[ссылки]]` между связанными страницами

**Один источник может затронуть 5–15 вики-страниц.**

### Операция 2: Query (Запрос)

Когда в вики накопилось 10+ страниц, начинай задавать вопросы:

- *«Какие три главных тренда в AI-агентах на основе всего в wiki/?»*
- *«Сравни статью X и статью Y. Где они расходятся?»*
- *«Напиши брифинг на 500 слов по теме [X] используя только материалы базы»*

LLM читает `wiki/index.md`, находит релевантные страницы, синтезирует ответ с цитатами.

> **Компаундинг:** хорошие ответы стоит сохранить в `outputs/` или вернуть в `wiki/` как новую страницу. Каждый вопрос делает следующий ответ лучше.

### Операция 3: Lint (Проверка здоровья)

Раз в месяц запускай проверку:

«Проверь всю `wiki/`. Найди противоречия между страницами. Найди темы упомянутые, но не раскрытые. Найди утверждения без источника. Предложи 3 новые статьи для заполнения пробелов.»

**Что проверяет Lint:**

- Противоречия между страницами
- Устаревшие утверждения
- Страницы-сироты (без входных ссылок)
- Концепции упомянутые, но без своей страницы
- Пропущенные перекрёстные ссылки

> **Почему это важно:** если LLM написал что-то неверное и ты сохранил это обратно — следующий ответ построится на ошибке. Lint ловит такие проблемы до того, как они накопятся. Это как code review для твоей базы знаний.

## 7. Способы загрузки данных

| Способ | Описание | Сложность |
|---|---|---|
| Obsidian Web Clipper | Кнопка в браузере → `.md` в `raw/articles/` | Легко |
| Копипаст | Создай `.md` файл вручную в `raw/` | Легко |
| Claude Code | «Сохрани эту статью в raw/» | Легко |
| Telegram Sync | Пересылка в бота → `raw/telegram/` | Средне |
| yt-dlp | YouTube транскрипты через CLI | Средне |
| X/Twitter архив | Запросить архив → распаковать в `raw/` | Средне |

### Obsidian Web Clipper (рекомендуемый)

1. Установи расширение из Chrome / Firefox / Safari Web Store
2. В настройках расширения укажи vault и папку `raw/articles/`
3. Нашёл интересную статью → нажал кнопку → файл в `raw/articles/`

> **Скачивание картинок:** Obsidian Settings → Hotkeys → «Download attachments» → привяжи к `Ctrl+Shift+D`. После клипа нажми хоткей — все картинки скачаются в `raw/images/`.

### CLI-инструменты

```bash
# YouTube транскрипты
yt-dlp --write-auto-sub --sub-lang ru \
    --skip-download -o "raw/notes/%(title)s" URL

# Резюме статьи по URL (steipete/summarize)
summarize https://example.com/article > raw/articles/2026-04-06-article.md
```

### Процесс после загрузки

После попадания нового файла в `raw/` говори LLM-агенту: «Обработай новые файлы в `raw/`» — и происходит операция Ingest.

## 8. Просмотр в Obsidian

### Graph View

Открой граф-вью (`Ctrl/Cmd+G`) — увидишь форму вики:

- **Хабы** — страницы с множеством входящих ссылок (ключевые концепции)
- **Сироты** — страницы без связей (нужно добавить ссылки или удалить)
- **Кластеры** — группы связанных тем

### Dataview

Плагин для запросов по frontmatter. Пример:

````
```dataview
TABLE type, domain, updated
FROM "wiki"
WHERE type != null
SORT updated DESC
```
````

### Поиск

До ~100 страниц достаточно `wiki/index.md` + стандартного поиска Obsidian.

Когда вики вырастет — подключи **qmd** (tobi): локальный поиск по markdown с BM25 + vector search и LLM re-ranking, всё на устройстве. Есть CLI и MCP-сервер.

## 9. Лучшие практики

### Делай

- **Загружай источники по одному** и участвуй в обсуждении с LLM — так вики получается качественнее
- **Сохраняй ценные ответы** обратно в wiki или outputs — это компаундит знания
- **Запускай Lint** хотя бы раз в месяц
- **Используй `[[вики-ссылки]]`** везде — они создают граф знаний
- **Держи CLAUDE.md актуальным** — это «мозг» системы, эволюционирующий со временем
- **Git** — вики это просто git-репо. Версионная история бесплатно

### Не делай

- **Не редактируй wiki вручную** — это работа LLM. Хочешь поправить — скажи LLM
- **Не модифицируй файлы в raw/** — они неизменяемые
- **Не усложняй** — плоские файлы и хорошая схема работают лучше навороченного стека плагинов
- **Не копи raw/ без обработки** — ценность в обработанной вики, а не в куче сырых файлов
- **Не фиксируйся на инструментах** — Obsidian с 47 плагинами = ловушка Notion

«I'm trying to keep it super simple and flat. It's just a nested directory of .md files.»
— Андрей Карпатый

## 10. Быстрый старт за 5 минут

1. **Установи Obsidian + Claude Code.** Если ещё не установлено.
2. **Создай папки.** `mkdir -p raw/articles raw/papers raw/notes wiki/_templates outputs`
3. **Создай CLAUDE.md.** Скопируй шаблон из раздела 5 или скажи Claude Code: «Настрой CLAUDE.md для LLM Knowledge Base».
4. **Найди интересную статью.** Сохрани её в `raw/articles/` (Web Clipper, копипаст или через Claude Code).
5. **Запусти Ingest.** Скажи Claude Code: «Обработай новый файл в `raw/articles/[имя-файла].md`».
6. **Открой Obsidian.** Перейди в `wiki/`, прочитай новые страницы, проверь graph view.
7. **Задай вопрос.** Спроси что-нибудь по теме статьи. Сохрани ценный ответ.
8. **Повтори.** Каждый цикл занимает 2–5 минут и делает базу знаний богаче.

## 11. FAQ

**Q: Можно использовать не Claude Code, а другой инструмент?**

A: Да. Cursor, Codex, любой LLM-агент с доступом к файлам. Паттерн не привязан к конкретному инструменту.

**Q: Что если вики станет слишком большой для контекста LLM?**

A: На ~100 страниц (~400 000 слов) Карпатый говорит что LLM справляется через `index.md` без RAG. При дальнейшем росте — подключить `qmd` для поиска.

**Q: Как это отличается от Notion/Obsidian с плагинами?**

A: Люди бросают вики потому что бухгалтерия (ссылки, обновления, консистентность) занимает больше времени чем ценность. LLM делает эту бухгалтерию бесплатно.

**Q: Нужен ли Obsidian?**

A: Нет. Подойдёт любой текстовый редактор. Obsidian удобен для визуализации графа и Dataview-запросов, но это опционально. Карпатый: «The AI doesn't care what app you open the files in.»

**Q: Можно ли использовать для команды?**

A: Да. Вики — это просто git-репо. Можно работать вместе, делать PR, branches. LLM обновляет вики из Slack, митингов, звонков.

**Q: А если LLM ошибётся?**

A: Для этого Lint. Запускай проверку раз в месяц. Плюс вики в git — можно откатить любые изменения.
