## Что такое Claude Code на самом деле

Март 2026 года. Если вы до сих пор копируете код в ChatGPT — пора серьёзно поговорить.

Я плотно работаю с Claude Code последние два месяца. Не от случая к случаю — каждый день. Прошёл курсы, получил сертификацию, строил проекты, ломал, чинил и строил снова. За это время я провёл в терминале с Claude больше времени, чем в разговорах с живыми людьми.

Это не очередная подборка «10 лучших AI-инструментов». Это всё, что я узнал на практике — функции, о которых не пишут, рабочие процессы, которые реально работают, ошибки, стоившие часов, и приёмы, сэкономившие дни.

Сохраняйте. Пригодится не раз.

### Главное заблуждение

Начну с того, что Claude Code — это не Copilot. Не чат-бот, куда вставляют код. Не автодополнение на стероидах.

Это **агент**. Автономный агент, который:

- читает всю вашу кодовую базу
- планирует подход
- редактирует файлы по всему проекту
- запускает тесты
- видит ошибки и исправляет их
- итерирует, пока задача не решена

Ключевое слово — **агентный** (agentic). Он работает в цикле: собирает контекст, действует, проверяет результат, повторяет. Вы говорите, чего хотите, — он сам разбирается, как этого достичь. Вы не пишете код вместе. Вы делегируете тому, кто действительно понимает прочитанное.

Claude Code живёт в терминале. Это не ограничение — в этом суть. Терминал — самый мощный интерфейс на вашей машине. Claude Code встречает вас там.

### Где он работает

Claude Code — давно не просто CLI-утилита. Он везде:

- **Терминал**: исходный вариант. Полная мощность. Самый быстрый. Здесь происходит магия.
- **Расширение VS Code**: инлайн-диффы, @-упоминания, боковая панель для разговоров. Для тех, кто живёт в редакторе.
- **Плагин JetBrains**: то же самое для пользователей IntelliJ.
- **Десктоп-приложение (Claude Cowork)**: визуальный просмотр диффов, несколько сессий, запланированные задачи. Терминал не нужен. Вариант для тех, кто не работает с командной строкой.
- **Веб-приложение (claude.ai/code)**: без локальной установки. Запускайте длинные задачи из любого места. Откройте браузер и работайте.
- **Мобильный доступ через remote control**: начните сессию на ноутбуке, управляйте с телефона.
- **Slack**: упомяните @Claude в рабочем пространстве, превращайте issues в PR прямо из чата.
- **GitHub Actions**: напишите @claude в комментарии к PR — он ответит реальными изменениями кода.
- **GitLab CI/CD**: та же концепция для команд на GitLab.

Функция remote control недооценена и мало обсуждается. Вы на обеде, приходит сообщение в Slack о баге, достаёте телефон, говорите Claude исправить. Он работает на вашей машине дома. Вы просматриваете дифф на телефоне. Одобряете. Готово.

## Модели

Claude Code работает на семействе моделей Anthropic. Знать, какую когда использовать — половина успеха:

- **Claude Opus 4.6**: тяжёлая артиллерия. Лучшие рассуждения. Для сложных архитектурных решений, хитрого дебаггинга, рефакторингов на 10+ файлов. Когда нужно по-настоящему думать.
- **Claude Sonnet 4.6**: рабочая лошадка. Модель по умолчанию. Лучший баланс скорости, стоимости и качества для повседневного кодинга. Ваш основной инструмент.
- **Claude Haiku**: спринтер. Дешёвый и быстрый. Хорош для простых задач и субагентов. Не стоит его недооценивать для быстрых вопросов.

Переключение моделей в любой момент:

```bash
/model opus
```

— переключение посреди сессии, когда упёрлись в сложную проблему.

```bash
claude --model opus
```

— запуск всей сессии с opus.

Вот что я усвоил на собственных ошибках: **соразмеряйте усилия с задачей**. Не тратьте токены opus на переименование переменной. Не используйте haiku для переделки системы авторизации. В первую неделю я потратил около $200, используя opus для всего подряд, потому что он «казался умнее». Sonnet справляется с 90% задач одинаково хорошо.

### Глубина размышлений

Глубину размышлений можно регулировать независимо от модели:

- `/effort low` — быстрые ответы, минимум рассуждений. Для «какой синтаксис у X?»
- `/effort high` — глубокий анализ, расширенное мышление. Для дебаггинга.
- `/effort max` — полная мощность. Тратит больше токенов, но ловит граничные случаи, которые пропустит человек. Для всего, что пойдёт в продакшен.

## Папка .claude — центр управления проектом

Здесь большинство людей теряют 80% пользы. Папка `.claude` — не чёрный ящик. Это центр управления поведением Claude в вашем проекте.

Важный момент, о котором мало кто знает: **существуют две директории .claude, а не одна**.

Первая — внутри проекта (коммитится в git, расшаривается с командой). Вторая — в `~/.claude/` (личные настройки, локальное состояние машины, история сессий).

### CLAUDE.md — самый важный файл

При запуске сессии Claude Code первым делом читает CLAUDE.md. Он загружается прямо в системный промпт. Всё, что там написано, Claude выполняет. Каждую сессию. Стабильно.

Если вы указали всегда писать тесты перед реализацией — он будет. Если написано «никогда не используй console.log, используй кастомный логгер» — он будет следовать этому каждый раз.

**Что писать:**
- Команды сборки, тестов и линтера (`npm run test`, `make build`)
- Ключевые архитектурные решения («мы используем монорепо с turborepo»)
- Неочевидные подводные камни («strict mode в TypeScript включён, неиспользуемые переменные — ошибка»)
- Соглашения по импортам, именованию, обработке ошибок
- Структура файлов и папок основных модулей

**Чего НЕ писать:**
- То, что относится к конфигу линтера или форматтера (этим занимается Prettier, не CLAUDE.md)
- Полную документацию, на которую можно дать ссылку
- Длинные абзацы с теорией

**Держите до 200 строк**

Файлы длиннее начинают съедать слишком много контекста, и следование инструкциям ухудшается. Я убедился в этом на практике: мой CLAUDE.md на 400 строк наполовину игнорировался. Сократил до 150 — всё стало работать лучше.

**Пример хорошего CLAUDE.md:**

```
Project: Acme API

Commands
npm run dev          # Start dev server
npm run test         # Run tests (Jest)
npm run lint         # ESLint + Prettier check
npm run build        # Production build

Architecture
- Express REST API, Node 20
- PostgreSQL via Prisma ORM
- All handlers live in src/handlers/
- Shared types in src/types/

Conventions
- Use zod for request body validation
- Return shape is always { data, error }
- Never expose stack traces to the client
- Use the logger module, not console.log

Watch out for
- Tests use a real local DB, not mocks. Run `npm run db:test:reset` first
- Strict TypeScript: no unused imports, ever
```

20 строк. Claude получает всё необходимое для продуктивной работы без постоянных уточнений.

### CLAUDE.local.md — личные настройки

Иногда у вас есть предпочтения, которые касаются только вас, а не команды. Может, вы предпочитаете другой тест-раннер. Может, хотите, чтобы Claude открывал файлы определённым образом.

Создайте `CLAUDE.local.md` в корне проекта. Claude читает его вместе с основным CLAUDE.md, и он автоматически попадает в `.gitignore` — ваши личные настройки никогда не попадут в репозиторий.

### Папка rules/ — модульные инструкции, которые масштабируются

CLAUDE.md отлично работает для одного проекта. Но когда команда растёт, вы получаете CLAUDE.md на 300 строк, который никто не поддерживает и все игнорируют.

Папка `rules/` решает эту проблему.

Каждый markdown-файл внутри `.claude/rules/` загружается вместе с CLAUDE.md автоматически:

```
.claude/rules/
-- code-style.md
-- testing.md
-- api-conventions.md
-- security.md
```

Каждый файл остаётся фокусированным. Ответственный за API-соглашения редактирует `api-conventions.md`. Ответственный за стандарты тестирования — `testing.md`. Никто не затирает чужие правки.

Настоящая сила — в **правилах с привязкой к путям**. Добавьте YAML-заголовок, и правило активируется только когда Claude работает с подходящими файлами:

```yaml
---
paths:
  - "src/api/**/*.ts"
  - "src/handlers/**/*.ts"
---
API Design Rules
All handlers return { data, error } shape
Use zod for request body validation
Never expose internal error details to clients
```

Claude не загрузит это правило, когда редактирует React-компонент. Оно срабатывает только при работе внутри `src/api/` или `src/handlers/`. Чисто, фокусированно, эффективно.

## Функции, которые изменили мой подход к работе

### Редактирование нескольких файлов

Здесь Claude Code оставляет всё остальное далеко позади. Именно эта функция заставила меня отказаться от других инструментов для серьёзной работы.

```bash
claude "refactor the authentication system to use JWT tokens instead of sessions"
```

Он:
- найдёт каждый файл, связанный с авторизацией
- обновит middleware
- изменит маршруты
- перепишет тесты
- обновит конфигурацию
- исправит импорты

В 15+ файлах за одну сессию. Показывает диффы для ревью перед применением. Вы одобряете, отклоняете или просите изменений по каждому файлу.

Я рефакторил целое Express-приложение с колбэков на async/await за одну сессию. 23 файла. Каждый — корректно. Попробуйте сделать это с автодополнением по Tab.

### Git-интеграция

Claude Code нативно работает с git и пишет коммит-сообщения лучше большинства людей:

```bash
claude "commit my changes with a descriptive message"
claude "create a PR for this feature"
claude "help me resolve these merge conflicts"
claude "show me what changed in the last 5 commits and explain the impact"
```

Он пишет нормальные коммит-сообщения — не «fixed stuff», а описания того, что изменилось и почему. Создаёт PR с резюме и планом тестирования. Разрешает merge-конфликты, действительно понимая код обеих сторон, а не выбирая одну наугад.

### Агентный цикл: написать, протестировать, починить, повторить

Claude не просто пишет код — он его запускает:

пишет функцию → запускает тесты → видит ошибку → читает лог → исправляет баг → снова запускает тесты → всё зелёное

Этот цикл — вся суть. Это не «вот код, удачи». Это «я написал, протестировал, поймал граничный случай, о котором вы не подумали, и всё проходит». Я видел, как он ловит баги в собственном коде, которые я бы не заметил часами.

### Субагенты — недооценённая функция

Мне понадобилось 3 недели, чтобы начать пользоваться ими нормально, и я жалею о каждом потерянном дне.

Claude может порождать специализированных субагентов для изолированной обработки задач:

- **Explore-агент**: только чтение, быстрый, использует Haiku. Идеален для «разберись, как работает этот модуль».
- **Plan-агент**: анализирует перед реализацией. Принуждает «думать перед тем, как писать».
- **General-агент**: сложные многошаговые задачи в чистом контексте.

Почему это важно: когда вы просите Claude запустить полный набор тестов — сотни строк вывода pass/fail засоряют контекстное окно. Основной разговор загрязняется шумом. Субагенты работают изолированно. Они делают грязную работу, сжимают результаты и возвращают чистое резюме.

### Кастомные агенты

Можно создавать собственных агентов для специфических рабочих процессов:

`.claude/agents/security-reviewer.md`

```yaml
---
name: security-reviewer
description: Expert code reviewer for security vulnerabilities.
  Use PROACTIVELY when reviewing PRs or before deployments.
model: sonnet
tools: Read, Grep, Glob
---
You are a senior security engineer. When reviewing code:
- Flag bugs, not just style issues
- Check for SQL injection and XSS risks
- Look for exposed credentials or secrets
- Check authentication and authorization gaps
- Note performance concerns only when they matter at scale
```

Теперь Claude автоматически делегирует ревью безопасности вашему кастомному агенту. Или вызовите его явно: `/security-reviewer`.

Поле `tools` намеренно ограничено — аудитору безопасности нужны только Read, Grep и Glob. Ему незачем записывать файлы. Поле `model` позволяет использовать Haiku для дешёвых задач чтения, а Opus — для задач, требующих глубокого анализа.

Личные агенты размещаются в `~/.claude/agents/` и доступны во всех проектах.

## MCP — Model Context Protocol

Здесь Claude Code превращается из «хорошего помощника для кодинга» в «оркестратор всего рабочего процесса».

**MCP позволяет Claude подключаться к внешним инструментам и сервисам:**

- **GitHub**: поиск по репозиториям, чтение issues, управление PR, ревью кода
- **Slack**: чтение каналов, отправка сообщений, ответы в тредах
- **PostgreSQL/MySQL**: запросы к базе данных напрямую
- **Jira**: обновление тикетов, смена статусов
- **Figma**: чтение дизайнов (да, серьёзно)
- **Puppeteer/Playwright**: автоматизация браузера
- **Sentry**: мониторинг ошибок
- **Notion**: чтение и редактирование документов

Конфигурация в `.mcp.json` в корне проекта:

```json
{
  "mcpServers": {
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "your_token"
      }
    }
  }
}
```

Теперь можно делать так:

```bash
claude "find all open bugs in our github repo tagged 'critical' and summarize them"
```

```bash
claude "query the database for users who signed up this week
        and post the count to #metrics on slack"
```

```bash
claude "check sentry for the top 5 errors this week
        and create github issues for each one"
```

Один инструмент, подключённый ко всему. У меня Claude вытягивает GitHub issues, читает базу данных и постит резюме в Slack в одном разговоре. Без переключения вкладок. Без потери контекста. Без копирования между инструментами.

## Хуки — детерминированная автоматизация

Инструкции в CLAUDE.md — это рекомендации. Claude следует им в большинстве случаев, но не всегда. Нельзя полагаться на языковую модель в том, что она *всегда* запустит линтер. *Всегда* отформатирует код. *Всегда* проверит опасные команды.

Хуки делают это поведение **детерминированным**. Это обработчики событий, которые срабатывают автоматически в определённых точках рабочего процесса Claude. Ваш shell-скрипт выполняется каждый раз, без исключений.

### События

- **PreToolUse**: срабатывает перед запуском любого инструмента. Защитный барьер. Блокируйте опасные команды здесь.
- **PostToolUse**: срабатывает после успешного выполнения инструмента. Автоформатирование, автолинт, автовалидация.
- **Stop**: срабатывает, когда Claude завершает задачу. Контроль качества. «Тесты должны пройти, прежде чем ты закончишь».
- **UserPromptSubmit**: срабатывает при нажатии Enter. Валидация промпта.
- **Notification**: десктопные уведомления, когда Claude требуется ваше внимание.

### Пример конфигурации хуков

Автоформатирование каждого файла, который Claude трогает, и блокировка опасных bash-команд:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write 2>/dev/null"
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/bash-firewall.sh"
          }
        ]
      }
    ]
  }
}
```

Скрипт bash-firewall читает команду из stdin, проверяет на опасные паттерны (`rm -rf`, `git push --force`, `DROP TABLE`) и выходит с кодом 2 для блокировки. Код 0 = пропустить. Код 1 = предупредить, но продолжить. Код 2 = заблокировать полностью.

Хук Stop, запускающий `npm test` и возвращающий код 2 при ошибке, не даст Claude объявить «готово», пока тесты не станут зелёными.

Хуки не перезагружаются на лету — перезапустите сессию после изменений. Они работают с полными правами пользователя, поэтому экранируйте переменные shell и валидируйте JSON-ввод.

## Скиллы — переиспользуемые рабочие процессы

Скиллы — функция, после которой я понял, насколько глубока кроличья нора.

Скилл — это рабочий процесс, который Claude может вызвать сам на основе контекста или который вы запускаете слэш-командой. Скиллы отслеживают разговор и действуют в нужный момент.

### Структура

Каждый скилл живёт в своей поддиректории с файлом SKILL.md:

```
.claude/skills/
-- security-review/
   -- SKILL.md
   -- DETAILED_GUIDE.md
-- deploy/
   -- SKILL.md
   -- templates/
      -- release-notes.md
```

SKILL.md использует YAML-заголовок для описания условий активации:

```yaml
name: security-review
description: Comprehensive security audit. Use when reviewing code
  for vulnerabilities, before deployments, or when the user
  mentions security.
allowed-tools: Read, Grep, Glob

Analyze the codebase for security vulnerabilities:
- SQL injection and XSS risks
- Exposed credentials or secrets
- Insecure configurations
- Authentication and authorization gaps

Report findings with severity ratings and specific remediation
steps. Reference @DETAILED_GUIDE.md for our security standards.
```

Когда вы говорите «проверь этот PR на безопасность», Claude читает описание, распознаёт совпадение и вызывает скилл автоматически. Или вызовите напрямую: `/security-review`.

Ключевое отличие от команд: **скиллы могут включать сопутствующие файлы**. Ссылка `@DETAILED_GUIDE.md` подтягивает документ, лежащий рядом с SKILL.md. Команды — это одиночные файлы. Скиллы — пакеты.

Скилл `/last30days`, ставший вирусным в X — наглядный пример. Кто-то создал скилл, который сканирует Reddit и X за последние 30 дней по любой теме, синтезирует находки сообщества и выдаёт готовые промпты. Набираете `/last30days prompting techniques for legal questions` — получаете фреймворки, которые реально используют юристы и опытные пользователи.

Личные скиллы размещаются в `~/.claude/skills/` и доступны во всех проектах.

**Прежде чем отпустить Claude на большой рефакторинг:**

1. Нажмите Shift+Tab дважды для входа в режим планирования
2. `claude "analyze the database layer and propose a migration to TypeORM"`

Claude исследует кодовую базу, читает релевантные файлы, анализирует архитектуру — и представляет план. Без изменений. Ничего не модифицировано. Только анализ и предложенный подход.

Вы проверяете. Корректируете. Задаёте вопросы. Ищете слабые места. Когда уверены — одобряете, и он реализует.

Это предотвращает момент «Claude переписал весь проект, а я не просил». Поверьте, этот момент не из приятных. Сначала план, потом реализация. Всегда.

## Система памяти

Claude запоминает вещи между сессиями. Автоматически.

Поправьте его один раз — «не используй классовые компоненты в этом проекте, мы используем хуки» — и он сохранит предпочтение. В следующей сессии он уже знает. Не приходится повторяться.

Управление через `/memory`. Хранится в `~/.claude/projects/<project>/memory/`.

Комбинация CLAUDE.md (знания команды) + автоматическая память (личное обучение) означает, что **Claude накапливает опыт**. Чем больше вы его используете, тем лучше он работает. Claude первой недели и Claude восьмой недели на одном проекте — два разных инструмента.

## Рабочие процессы для продвинутых

Паттерны, которые превратили меня из «пользователя Claude Code» в «опасного пользователя Claude Code». Ничего из этого нет в документации.

### Техника «Проинтервьюируй меня»

Начинаете сложный проект? Не пишите огромный промпт. Не пытайтесь продумать всё заранее. Просто скажите:

```bash
claude "interview me in detail about what i want to build"
```

Пусть Claude задаёт ВАМ вопросы. Он спросит про стек, требования, граничные случаи, существующий код, цели деплоя, типы пользователей — вещи, которые вы забыли бы упомянуть в промпте. 10 минут диалога дают Claude лучший контекст, чем промпт на 500 слов.

Я использую это для каждой новой фичи. Для каждой.

### Разделение «Исследование → Реализация»

Никогда не давайте Claude реализовывать то, чего он сначала не понял.

**Фаза 1:** Shift+Tab дважды → режим планирования → `"explore the codebase and understand how payments work"`

Проверьте анализ. Убедитесь, что он действительно разобрался.

**Фаза 2:** `"implement subscription billing based on your analysis"`

Разница в качестве между «просто сделай» и «сначала разберись, потом делай» — колоссальная. Особенно на legacy-кодовых базах, где ничего не лежит там, где ожидаешь.

### Параллельная работа с worktree

```
Терминал 1: claude --worktree auth-feature
Терминал 2: claude --worktree billing-feature
```

Каждый получает изолированную git-ветку. Две фичи разрабатываются одновременно двумя отдельными сессиями Claude. Мержите, когда готово.

Иногда я запускаю 3 worktree. Фича авторизации в терминале 1, API-эндпоинт в терминале 2, набор тестов в терминале 3. Всё параллельно. Всё изолировано.

### Управление контекстом

**Это навык #1, который отличает хороших пользователей Claude Code от отличных.**

Контекстное окно — около 200K токенов. Кажется много. Но когда вы глубоко в сессии, оно быстро заканчивается. По мере заполнения:
- сначала очищается вывод старых инструментов
- разговор автоматически сжимается
- ранние инструкции могут быть потеряны

**Управляйте активно:**
- `/compact` — ручное сжатие разговора для экономии места
- `/clear` — чистый старт между несвязанными задачами
- `/cost` — посмотреть, что съедает токены и где
- Субагенты для объёмных операций (вывод тестов, анализ логов, чтение больших файлов)
- Повторяющиеся инструкции — в CLAUDE.md, не в разговор

**Золотое правило:** после двух неудачных попыток — остановитесь. Не продолжайте. `/clear` и напишите лучший начальный промпт с чётким промптом почти всегда работает лучше, чем загрязнённый контекст с неудачными подходами. Я усвоил это ценой примерно 6 часов потраченного времени.

### Префикс !

Введите `!` перед любой командой, чтобы выполнить её прямо в shell без Claude:

```bash
!git status
!npm test
!cat src/config.ts
```

Вывод автоматически добавляется в контекст Claude. Удобно для быстрой демонстрации Claude текущего состояния без запроса на выполнение команды и ожидания разрешений.

### Внешний редактор для длинных промптов

`Ctrl+G` открывает системный редактор (vim, VS Code — что настроено). Пишите сложные многострочные промпты с подсветкой синтаксиса. Сохраните и закройте — отправится в Claude.

Гораздо удобнее, чем набирать абзац в однострочном терминальном вводе. Я использую это для каждого промпта длиннее двух предложений.

### Двойной Escape — откат назад

Этот приём мало кто упоминает.

Claude пошёл не туда? Двойное нажатие Escape вызывает меню отката. Можно отменить последнее действие, вернуться дальше или «сжать отсюда» — это сжимает неудачную попытку, сохраняя полезный контекст. Гораздо лучше `/clear`, потому что вы не теряете всё — только ошибку.

### Режимы разрешений

Claude Code имеет разные уровни автономности:

Начните с режима по умолчанию. Переключитесь на `acceptEdits`, когда доверяете (у меня это заняло около недели). Используйте план для исследования перед большими рефакторингами.

**Белые и чёрные списки команд** — в `.claude/settings.json`:

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Read",
      "Write",
      "Edit"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./.env.*)"
    ]
  }
}
```

Белый список выполняется без вопросов. Чёрный — блокируется полностью. Всё, что не в обоих списках, Claude спрашивает разрешения. Эта золотая середина намеренна. Защитная сетка без микроменеджмента каждой команды.

## Claude Code vs Cursor vs Copilot

Честное сравнение. Без фанатизма. Только личный опыт.

**Claude Code**: нативный в терминале, по-настоящему агентный. Читает всю кодовую базу. Автономные мультифайловые изменения. Лучше всего для: сложных рефакторингов, архитектурных решений, изменений по всему проекту, дебаггинга запутанных проблем, всего, что затрагивает больше 3 файлов. Ощущается как спокойный senior-инженер, который никогда не устаёт и не осуждает ваш код.

**Cursor**: нативный в IDE (форк VS Code). Лучший повседневный инструмент для автодополнения по Tab и инлайн-подсказок. Более тесная обратная связь в редакторе. Менее автономный, но быстрее для мелких правок. Лучше всего для: быстрых правок, инлайн-подсказок, поддержания потока при написании кода.

**GitHub Copilot**: подход через плагин. Лучший бесплатный тариф ($10/мес). Самый безопасный корпоративный выбор. Агентные возможности растут, но пока отстают от Claude Code и Cursor.

Многие не понимают: **не обязательно выбирать одно**.

Комбинированная стратегия реальна, и большинство опытных пользователей так и делают. Cursor — для повседневного автодополнения и инлайн-правок. Claude Code — для тяжёлой работы: рефакторинги, новые фичи, дебаггинг, архитектурные решения.

Данные это подтверждают: опытные разработчики используют в среднем **2,3 AI-инструмента для кодинга**. Дело не в замене одного другим. Дело в правильном инструменте для правильной задачи.

Claude Code вышел на позицию **#1 самого любимого AI-инструмента для кодинга** менее чем за год. 46% разработчиков назвали его фаворитом в опросах начала 2026 года. 95% разработчиков используют AI-инструменты еженедельно. 75% используют AI более чем для половины кодинга.

Правила игры изменились. Насовсем. Вопрос не в том, использовать ли AI-инструменты для кодинга. Вопрос в том, как быстро вы научитесь с ними работать.

## Экосистема Claude Code

Сообщество создало огромное количество инструментов поверх Claude Code. Вот те, что стоит знать:

- **Superpowers (obra/superpowers)**: полная методология разработки для AI-агентов. 117K+ звёзд на GitHub. Меняет подход агента к написанию кода — заставляет его сначала провести брейнсторм, создать детальный план реализации, запустить разработку через субагентов и провести двухэтапное код-ревью перед завершением. Если Claude Code иногда выдаёт «спагетти» — это лекарство.

- **GSD — Get Shit Done (gsd-build/get-shit-done)**: система мета-промптинга и управления контекстом для Claude Code. Лёгкая, но мощная. Решает проблему деградации контекста.

- **awesome-claude-code**: курируемая сообществом коллекция лучших ресурсов, скиллов, агентов и MCP-серверов. Стартовая точка для знакомства с экосистемой.

- **Скилл /last30days**: тот самый вирусный скилл. Сканирует Reddit, X, YouTube, Hacker News и веб за последние 30 дней по любой заданной теме. Синтезирует знания сообщества в готовые промпты. Набираете `/last30days cold email frameworks` — он находит фреймворк 3 Ps, ADA, триггеры на основе намерений — вещи, которые вы не найдёте сами. Затем пишет готовые промпты на основе того, что реально работает. Open source, лицензия MIT.

- **Claude How-To**: визуальное руководство по освоению Claude Code с примерами. Лучший ресурс для визуалов, предпочитающих структурированные туториалы.

- **Claude Mem**: слой персистентной памяти. Если встроенной системы памяти недостаточно для вашего рабочего процесса.

- **UI UX Pro Max**: дизайн-ориентированный инструментарий для Claude Code. Когда нужно, чтобы Claude заботился о том, как вещи выглядят, а не только о том, как работают.

- **n8n-MCP**: подключает Claude Code к автоматизации n8n. Если вы уже используете n8n — это очевидная интеграция.

Экосистема растёт быстро. Новые инструменты каждую неделю. Здесь перечислены те, которые я реально использовал или видел в деле.

## Типичные ошибки (я совершил все)

### 1. Роман вместо CLAUDE.md

Держите до 200 строк. Конкретно. Применимо. Если Claude игнорирует инструкции — вероятно, CLAUDE.md слишком длинный. У меня был файл на 400 строк — практически манифест. Сократил до 150 — всё наладилось.

### 2. Нет /clear между задачами

Разговор о починке CSS-бага не содержит полезного контекста для нового API-эндпоинта. Остаточный контекст активно мешает. Очищайте. Начинайте заново. Каждый раз.

### 3. Борьба вместо перезапуска

После двух неудачных попыток контекст загрязнён ошибочными подходами. Claude начинает ссылаться на собственные ошибки. `/clear` и пишите лучший промпт. Свежий контекст почти всегда справляется. Однажды я потратил 6 часов, прежде чем это усвоил.

### 4. Игнорирование субагентов

Запуск полного набора тестов в основном разговоре засоряет контекст сотнями строк вывода. Делегируйте субагенту. Держите основной разговор чистым и сфокусированным на работе.

### 5. Opus для всего

Sonnet справляется с 90% задач одинаково хорошо. Opus — для тех 10%, которым реально нужны глубокие рассуждения: сложная архитектура, хитрый дебаггинг, тонкие логические ошибки. Ваш кошелёк скажет спасибо. Мой сказал.

### 6. Пропуск режима планирования для больших изменений

Всегда планируйте, если затронуты больше 3 файлов. Всегда. 5 минут на проверку плана экономят 2 часа на исправлениях, когда Claude пошёл не туда.

### 7. Пассивное отношение к контексту

Контекстное окно — ресурс. Относитесь соответственно. `/compact`, когда разговор затянулся. Субагенты для объёмных операций. CLAUDE.md для инструкций вместо повторения каждую сессию. Люди, которые управляют контекстом хорошо, получают кардинально лучшие результаты.
